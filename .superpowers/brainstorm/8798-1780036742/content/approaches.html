设置页实现方案

三种架构方向，各有侧重——选你最认可的

A

最小改动方案（推荐）

新增 /settings 路由页，前端存主题色到 localStorage，AI 配置（Key + Base URL）通过新增 Go API 端点 PUT /api/v1/settings 保存到数据库（加密存储）。新增一个 settings-store.ts 管理前端状态，主题色通过操作 CSS 变量实现实时预览。后端新增 User Settings 表。

优点

  • 改动范围清晰
  • 主题响应快（纯前端）
  • Key 安全（只走后端）

缺点

  • 主题色无法多设备同步
B

全量后端方案

所有设置（包括主题色）都存数据库，登录后从 GET /api/v1/settings 拉取，写入 CSS 变量。好处是多设备同步，但主题色需等 API 响应才生效，可能出现短暂白屏/闪烁（FOUC）。

优点

  • 设置跨设备一致

缺点

  • 主题加载有延迟
  • 后端改动更大
C

混合方案

主题色存 localStorage（即时响应），AI Key + Base URL 存后端加密数据库，两套存储分开管理。同方案 A，但更明确区分"外观设置"和"服务配置"两个不同的关注点。

优点

  • 最佳用户体验
  • 安全性有保障

缺点

  • 两套持久化逻辑
